IT Admin Guide: Infrastructure & HA

Hyper-V, Oracle Database & Monitoring System

🛠️ 1. การติดตั้ง Hyper-V แบบละเอียด

ตรวจสอบใน BIOS ของ Server ว่าเปิด Virtualization (VT-x / AMD-V) เรียบร้อยแล้ว

  1. เปิด Server Manager บน Windows Server
  2. มุมขวาบนคลิก Manage > เลือก Add roles and features
  3. กด Next ไปเรื่อยๆ จนถึงหน้า Server Roles ให้ติ๊กถูกที่ช่อง Hyper-V
  4. หน้า Virtual Switches: เลือกว่าจะให้ VM ใช้การ์ดแลน (NIC) ใบไหนเพื่อออกเน็ต (หาก Server มีหลายพอร์ต ให้เลือกพอร์ตที่ต่อสายแลน External ไว้)
  5. หน้า Migration: ติ๊กถูกที่ "Allow this server to send and receive live migrations..."
  6. หน้า Default Stores: สำคัญมาก! ให้เปลี่ยน Path ที่เก็บไฟล์ VHDX และ Configuration ไปไว้ที่ Drive อื่นที่ไม่ใช่ Drive C: (เช่น D:\Hyper-V)
  7. กด Install และ Restart เครื่อง 1 รอบ
💡 Pro Tip: แนะนำให้ดาวน์โหลดและติดตั้ง Windows Admin Center (WAC) ซึ่งเป็น Web GUI ฟรีของ Microsoft มาใช้จัดการ Hyper-V แทน Server Manager จะช่วยให้ดูหน้าจอตั้งค่า Network และจัดการ VM ได้ง่ายขึ้นมาก

💾 2. วิธีการ Backup VM (ระดับ OS)

วิธีที่ 1: ใช้ Windows Server Backup (ฟรีในตัว)

  1. ติดตั้ง Feature: Windows Server Backup ผ่านหน้า Server Manager
  2. เปิดโปรแกรม > เลือก Backup Schedule...
  3. เลือก Custom > Add Items > เลือกที่โฟลเดอร์ Hyper-V แล้วติ๊กเลือก VM ที่ต้องการ Backup
  4. กำหนดเวลา (ควรเป็นช่วงดึก) และเลือก Destination ให้เซฟลง External HDD หรือ Network Share

วิธีที่ 2: ใช้ Veeam Community Edition แนะนำที่สุด

  • ติดตั้ง Veeam Backup & Replication (สามารถโหลดเวอร์ชัน Community มาใช้ได้ฟรีสำหรับ 10 Workloads)
  • เพิ่ม Hyper-V Server ของคุณเข้าไปในระบบของ Veeam
  • สร้าง Backup Job และเลือก VM ที่ต้องการ
  • จุดเด่น: แนะนำให้เปิดฟีเจอร์ Application-aware processing ตอนสร้าง Job ด้วย เพื่อให้ระบบ Backup ทำงานประสานกับ Database ภายใน VM ได้อย่างสมบูรณ์ (VSS Integration)

🔄 3. การทำ Replication (DR Site ระดับ VM)

คือการส่งสำเนา VM ทั้งก้อนจาก Host ต้นทางไป Host ปลายทาง เพื่อสลับไปเปิดใช้งานเวลาเครื่องหลักพัง

Step 1: ตั้งค่า Replica Broker (ทำทั้งเครื่องหลักและเครื่องสำรอง)

  1. เปิด Hyper-V Manager > คลิกที่ Hyper-V Settings... ด้านขวามือ
  2. ไปที่เมนู Replication Configuration
  3. ติ๊ก Enable this computer as a Replica server
  4. เลือก Use Kerberos (HTTP) และในส่วน Authorization เลือก "Allow replication from the specified servers" ใส่ IP เครื่องตรงข้ามลงไป

Step 2: เริ่มทำ Replication บน VM (ทำที่เครื่องหลัก)

  1. คลิกขวาที่ VM เครื่องต้นทาง > เลือก Enable Replication...
  2. ใส่ชื่อหรือ IP ของ Replica Server (เครื่องปลายทาง)
  3. เลือก VHDX ที่ต้องการส่งไป
  4. ตั้งค่า Replication Frequency (มีให้เลือก 30 วินาที, 5 นาที, 15 นาที)
  5. เลือก "Send initial copy over the network" > กด Finish ระบบจะเริ่มโยนข้อมูลก้อนแรกทันที

🔌 4. แก้ปัญหา VM มองไม่เห็น Network

1. ปัญหาเน็ตหลุดสุ่มๆ หรือ Ping ไม่เจอ (ปิด VMQ) เจอบ่อยมาก

มักเกิดกับ Server ที่ใช้การ์ดแลนของ Broadcom ที่ฟีเจอร์ Virtual Machine Queue (VMQ) ทำงานผิดปกติ ให้แก้ดังนี้:

  1. คลิกขวาที่ VM ที่มีปัญหา > เลือก Settings...
  2. กางเมนู Network Adapter ออกมา > คลิกที่ Hardware Acceleration
  3. ด้านขวามือ เอาเครื่องหมายติ๊กถูกออก จากช่อง Enable virtual machine queue

2. การใส่ VLAN ID ให้กับ VM

  • หากองค์กรใช้ VLAN ให้เข้าไปที่ Settings ของ VM > Network Adapter > ติ๊กช่อง Enable virtual LAN identification แล้วใส่เลข VLAN ID ให้ถูกต้อง

3. MAC Address Spoofing

  • หาก VM นั้นทำหน้าที่เป็น Load Balancer, HAProxy หรือรัน Docker/Kubernetes ซ้อนอยู่ข้างใน ต้องเข้าไปกางเมนู Advanced Features ของ Network Adapter แล้วติ๊กเปิด Enable MAC address spoofing ด้วย

🛢️ 5. แนวทางการตั้งค่า VM สำหรับ Oracle Database

การนำ Hyper-V มารัน Database ระดับ Enterprise อย่าง Oracle (เช่น 19c) มีกฎเหล็กเพื่อป้องกันระบบหน่วงหรือล่มดังนี้:

1. การตั้งค่า Storage (VHDX) บังคับทำ

  • ต้องสร้างไฟล์ VHDX เป็นแบบ "Fixed Size" (ขนาดคงที่) เท่านั้น ห้ามใช้ Dynamically expanding เด็ดขาด เพื่อให้ได้ความเร็ว I/O ของดิสก์สูงสุด และป้องกัน Disk Fragmentation
  • ควรแยก Drive ของ OS (C:) และ Drive ของ Database (เช่น D:, E:) ออกจากกัน คนละไฟล์ VHDX

2. การตั้งค่า Memory (RAM) บังคับทำ

  • ไปที่ Settings > Memory
  • เอาเครื่องหมายติ๊กถูกออก จากช่อง Enable Dynamic Memory
  • ใส่ Startup RAM เป็นแบบคงที่ (Static) เช่น ขั้นต่ำ 8192 MB (8GB) ขึ้นไปสำหรับ Production เพื่อให้ Oracle จองพื้นที่หน่วยความจำหลัก (SGA/PGA) ได้อย่างเสถียร ไม่โดน Windows ดึง RAM กลับ

3. CPU (vCPU)

  • ขั้นต่ำ 2 Core (แนะนำ 4-8 Core ขึ้นไปสำหรับระดับ Production)

🛡️ 6. การ Backup Oracle Database (ฉบับฟรี)

⚠️ ข้อควรระวังสูงสุด: Veeam/Hyper-V Backup กับ Oracle
ห้ามใช้การทำ Snapshot หรือการสั่ง Backup ระดับ VM (จากข้อ 2) ถ่าย Backup VM ที่กำลังรัน Oracle อยู่โดยตรง (เว้นแต่จะตั้งค่า VSS Integration ถูกต้อง 100%) เพราะอาจทำให้ไฟล์ Database "Corrupt" ได้
👉 วิธีที่ถูกต้อง: ให้รัน RMAN ถ่าย Backup ลงเป็นไฟล์ในเครื่อง VM ก่อน แล้วค่อยให้ Veeam/Windows Backup มาดูดโฟลเดอร์นั้นไปเก็บอีกชั้นหนึ่ง

RMAN (Recovery Manager) แนะนำที่สุด

เป็นเครื่องมือฟรีที่มีใน Oracle ทุกรุ่น รองรับการทำ Full, Incremental และ Hot Backup (Backup ได้โดยไม่ต้องปิด Database)

# เปิด Command Prompt เข้าสู่ระบบ RMAN
rman target /

# 1. ตั้งค่าให้เซฟ Controlfile อัตโนมัติทุกครั้งที่มีการ Backup
CONFIGURE CONTROLFILE AUTOBACKUP ON;

# 2. สั่ง Backup ฐานข้อมูลทั้งหมดพร้อมไฟล์ Archive Log
BACKUP DATABASE PLUS ARCHIVELOG;

💡 แนะนำให้เขียนคำสั่งข้างต้นเป็นไฟล์ .bat หรือ .sh แล้วตั้ง Task Scheduler รันทุกคืนตอนตี 2

Oracle Data Pump (expdp/impdp)

เหมาะสำหรับการ Backup แบบ Logical (เอาเฉพาะโครงสร้าง หรือบาง Table) ดึงข้อมูลข้ามไปอีกเครื่อง

# โยน Schema ชื่อ HR ออกเป็นไฟล์ .dmp
expdp system/password schemas=HR directory=BACKUP_DIR dumpfile=hr_backup.dmp

🔗 7. เจาะลึกวิธี Replicate Oracle Database (ไม่มี Data Guard)

ฟีเจอร์ Oracle Data Guard ต้องเสียเงินซื้อ License แบบ Enterprise Edition หากองค์กรใช้ Standard Edition (SE2) เรามีทางเลือกฟรี 3 วิธีที่ดีที่สุด ดังนี้ครับ:

วิธีที่ 1: Log Shipping (Manual Standby) ประหยัดสุด ยอดนิยม

หลักการ: อาศัยการก๊อปปี้ไฟล์ Archive Log ที่เกิดใหม่จากเครื่อง Primary โยนข้าม Network ไปให้เครื่อง Standby รันคำสั่งอ่าน (RPO ข้อมูลหายประมาณ 5-15 นาที)

🚨 ข้อควรระวัง (จุดตายของวิธีนี้)
  • ORA-00257: หาก Drive ที่เก็บ Archive Log ฝั่ง Primary "พื้นที่เต็ม" Database จะหยุดชะงัก (Hang) ทันที ต้องทำ Housekeeping คอยลบไฟล์เก่าเสมอ
  • Log Gap: ห้ามไฟล์ระหว่างทางหายหรือข้ามลำดับ (Sequence) เพราะเครื่อง Standby จะหยุดซิงค์ข้อมูลต่อทันที

ขั้นตอนการทำ (Step-by-Step)

  1. [Primary] เปิดโหมดจัดเก็บ Log และแยก Drive: 
    SQL> SHUTDOWN IMMEDIATE;
SQL> STARTUP MOUNT;
SQL> ALTER DATABASE ARCHIVELOG;
SQL> ALTER SYSTEM SET log_archive_dest_1='LOCATION=E:\Oracle\Archive' SCOPE=BOTH;
SQL> ALTER DATABASE OPEN;
  2. [Standby] สร้าง Share Folder: ไปที่เครื่องสำรอง สร้างโฟลเดอร์ D:\Oracle_Archive_Standby แล้ว Share ให้เครื่องหลักมีสิทธิ์ Write เข้ามาได้
  3. [Primary] ตั้ง Script ส่งไฟล์ (Sync): สร้างไฟล์ sync_log.bat ใช้คำสั่ง Robocopy เพื่อความทนทานต่อเน็ตหลุด (ตั้ง Task Scheduler รันทุก 5 นาที หรือรันแบบ Monitor) 
    REM ส่งไฟล์ .arc โดย /Z สานต่อเมื่อเน็ตหลุด และ /MOT:5 มอนิเตอร์ไฟล์ใหม่ทุก 5 นาที
robocopy E:\Oracle\Archive \\DR-Server\Oracle_Archive_Standby *.arc /Z /MOT:5
  4. [Standby] สร้างเครื่องสำรองและรับข้อมูล:
    • นำก้อน Backup จากเครื่องหลักมา Restore ที่เครื่อง Standby ให้ถึงแค่สถานะ MOUNT
    • สร้าง Script รันลูปอัตโนมัติ เพื่อดึง Log เข้า Database ตลอดเวลา 
      SQL> RECOVER AUTOMATIC STANDBY DATABASE;
วิธีที่ 2: Oracle SEHA (High Availability) ระบบไม่ล่ม

หลักการ: ใช้เทคโนโลยี Cluster ทดแทน RAC โดยใช้ Server 2 เครื่องช่วยกันเฝ้า Storage ก้อนเดียวกัน ถ้าเครื่องหลักดับ เครื่องสำรองจะดึง Storage ไปรัน Service ต่อทันที (ข้อมูลไม่สูญหาย 100%) เหมาะกับ Oracle 19c SE2

ขั้นตอนการทำ

  1. สร้าง Shared Disk (ใน Hyper-V ใช้ VHDX แบบ Shared Virtual Hard Disk) Mount เข้า VM ทั้ง 2 เครื่อง
  2. ติดตั้ง Oracle Grid Infrastructure เลือกโหมด Standalone Server (Oracle Restart) เพื่อจัดการดิสก์ด้วย ASM (Automatic Storage Management)
  3. ติดตั้ง Oracle Database ลงบน ASM Disk ที่แชร์ไว้ตรงกลาง
  4. ใช้คำสั่ง srvctl add database เพื่อให้ Clusterware รู้จัก หาก VM เครื่อง A ร่วง ระบบ GI จะใช้เวลาประมาณ 1-2 นาทีในการดึง Service ไปเปิดที่เครื่อง B อัตโนมัติด้วย IP เดิม
วิธีที่ 3: Logical Replication (CDC - Change Data Capture) ยืดหยุ่นสูงสุด

หลักการ: ใช้ Open Source Tool เช่น SymmetricDS ดักจับการเปลี่ยนแปลงข้อมูล (Insert/Update) "ระดับแถว (Row)" จาก Redo Log แล้วยิงผ่าน HTTP ไปยัง Database ปลายทางแบบ Real-time

ขั้นตอนการทำ

  1. เปิด Supplemental Logging ที่เครื่องหลักเพื่อให้เก็บบันทึกข้อมูลการเปลี่ยนแปลงได้ครบถ้วน
  2. ติดตั้ง SymmetricDS Agent (Java) ไว้ที่เครื่อง Master และ Target
  3. ตั้งค่าไฟล์ engine.properties ระบุ DB URL, Username/Password
  4. กำหนด Routing ใน GUI หรือรัน Script ระบุว่า "ให้จับตาดูตาราง PATIENT_INFO ถ้ามีการแก้ ให้ยิงข้อมูลไปที่เครื่อง DR"
  5. จุดเด่น: เครื่อง Standby สามารถเปิดอยู่ในสถานะ Read-Write ได้ตลอดเวลา เหมาะมากสำหรับดึงข้อมูลไปทำ Dashboard, Web App หรือซิงค์ข้ามยี่ห้อ (Oracle ไป SQL Server/PostgreSQL) โดยไม่รบกวนประสิทธิภาพเครื่องหลัก

🏆 สรุป: เลือกวิธีไหนดีที่สุด?

  • Log Shipping: เหมาะสุดสำหรับคนงบจำกัด หรือเพิ่งเริ่มทำ DR Site ครั้งแรก ทำง่าย แต่อย่าลืมมอนิเตอร์เรื่องพื้นที่ดิสก์
  • SEHA: เหมาะสำหรับระบบ Hospital/Healthcare ที่ข้อมูลหายไม่ได้แม้แต่ Record เดียว แต่มีข้อแม้ว่าคุณต้องมีอุปกรณ์ SAN Storage ตรงกลางที่เสถียรมากๆ
  • CDC (SymmetricDS): เหมาะที่สุดหากคุณต้องการแยกระบบ Report ออกมา หรือมีแผนจะเอาข้อมูลไปเขียนเชื่อมกับ Google Apps Script / Web Application สมัยใหม่ครับ

📈 8. ระบบ Monitoring & การแจ้งเตือน (Alerting)

เพื่อป้องกันปัญหา "Archive Log เต็มจนระบบล่ม" หรือ "VM ดับโดยไม่มีคนรู้" การสร้างตาข่ายเฝ้าระวังคือสิ่งที่ IT Admin ขาดไม่ได้ครับ

1. ใช้ Checkmk มอนิเตอร์แบบรวมศูนย์

Checkmk (Raw/Community Edition) เป็น Open Source ที่เหมาะกับงานนี้ที่สุด (สามารถติดตั้งลงบน Ubuntu Server 22.04 LTS ได้ฟรี)

  1. Hyper-V & Windows: ติดตั้ง Checkmk Agent บน Windows Server แล้วนำไฟล์ Plugin hyperv_vms.ps1 ไปใส่ ระบบจะดึงสถานะของ VM ทุกตัว (CPU/RAM/State) ขึ้นมาโชว์บน Dashboard ทันที
  2. Oracle Database: ใช้ Plugin mk_oracle วางที่เครื่อง Database สร้าง User ให้สิทธิ์อ่าน ระบบจะตรวจสอบพื้นที่ Tablespace (ป้องกันฐานข้อมูลเต็ม), สถานะ Archive Log, และการรัน Backup ของ RMAN ให้แบบลงลึก
2. แจ้งเตือนเข้า LINE / Telegram ด้วย PowerShell

ไม่ต้องรอให้คนเข้าไปดู Dashboard คุณสามารถเขียน Script ให้ระบบยิงข้อความเข้ามือถือทีม IT เวลามีปัญหาฉุกเฉินได้ (ตั้ง Task Scheduler รันทุก 10 นาที)

ตัวอย่างโค้ด: เช็คพื้นที่ Archive Log แล้วยิง LINE Notify

# ใส่ Token ของ LINE Notify กรุ๊ป IT
$Token = "YOUR_LINE_NOTIFY_TOKEN_HERE"

# เช็คพื้นที่ Drive E: (ที่เก็บ Archive Log)
$Drive = Get-WmiObject Win32_LogicalDisk | Where-Object { $_.DeviceID -eq "E:" }
$FreeSpaceGB = [math]::Round($Drive.FreeSpace / 1GB, 2)
$TotalSpaceGB = [math]::Round($Drive.Size / 1GB, 2)
$PercentFree = [math]::Round(($FreeSpaceGB / $TotalSpaceGB) * 100, 2)

# ถ้าพื้นที่เหลือน้อยกว่า 20% ให้ส่งแจ้งเตือน
if ($PercentFree -lt 20) {
    $Message = "⚠️ [CRITICAL] พื้นที่ Archive Log (Drive E:) เหลือเพียง $PercentFree% ($FreeSpaceGB GB) ระวัง Database ล่ม กรุณา Clear Log ด่วน!"
    $Params = @{
        Uri = "https://notify-api.line.me/api/notify"
        Method = "Post"
        Headers = @{ "Authorization" = "Bearer $Token" }
        Body = @{ "message" = $Message }
    }
    Invoke-RestMethod @Params
}

💡 ประยุกต์ใช้: โค้ดแนวนี้สามารถนำไปแปลงเป็น Webhook สำหรับ Telegram Bot ได้เช่นกัน เพื่อแยกกลุ่มแจ้งเตือนระหว่าง "Admin" (ปัญหา Server) และ "Tech/Staff" (ปัญหาการใช้งานทั่วไป) ให้ชัดเจนขึ้นครับ

💻 9. PowerShell Commands (Cheat Sheet)

# 1. ตรวจสอบสถานะ VM ทั้งหมดในเครื่อง (ดูว่าตัวไหนปิดอยู่)
Get-VM | Select-Object Name, State, Uptime, CPUUsage

# 2. เริ่มต้นเปิด VM ทั้งหมดที่โดนปิดอยู่
Get-VM | Where-Object {$_.State -eq 'Off'} | Start-VM

# 3. ปิด VM อย่างปลอดภัย (ส่งคำสั่ง Shutdown ไปที่ OS ข้างใน)
Stop-VM -Name "Oracle_DB" -Force

# 4. ดูสถานะการทำ Replication (เช็ค Health ว่า DR Site ทำงานปกติไหม)
Get-VMReplication

# 5. ตรวจสอบความสมบูรณ์ของไฟล์ VHDX
Test-VHD -Path "D:\Hyper-V\Database.vhdx"

# 6. ตรวจสอบพื้นที่ว่างของฮาร์ดดิสก์ทั้งหมดในเครื่องแบบอ่านง่าย
Get-PSDrive C,D,E | Select-Object Name, @{n='Used(GB)';e={[math]::round($_.Used/1GB,2)}}, @{n='Free(GB)';e={[math]::round($_.Free/1GB,2)}}
💡 ข้อแนะนำทิ้งท้าย: ไม่ควรเก็บ Snapshot (Checkpoint) ของ VM ทิ้งไว้นานเกิน 24-48 ชั่วโมง เพราะนอกจากจะทำให้กินพื้นที่ฮาร์ดดิสก์แล้ว ไฟล์ VHDX จะแตกแขนง (Differencing Disk) ทำให้ความเร็ว I/O ของระบบ Oracle ลดลงอย่างหนักจน User บ่นว่าระบบช้าได้ครับ
