IT Admin Guide

Hyper-V & Oracle Database (Enterprise Free Edition)

🛠️ 1. การติดตั้ง Hyper-V แบบละเอียด

ตรวจสอบใน BIOS ของ Server ว่าเปิด Virtualization (VT-x / AMD-V) เรียบร้อยแล้ว

  1. เปิด Server Manager บน Windows Server
  2. มุมขวาบนคลิก Manage > เลือก Add roles and features
  3. กด Next จนถึงหน้า Server Roles ติ๊กถูกที่ช่อง Hyper-V
  4. หน้า Virtual Switches: เลือก Network Card (NIC) ที่จะใช้ออกเน็ต
  5. หน้า Default Stores: สำคัญมาก! เปลี่ยน Path เก็บ VHDX ไปไว้ Drive อื่นที่ไม่ใช่ C:
  6. กด Install และ Restart เครื่อง 1 รอบ

💾 2. วิธีการ Backup VM

วิธีใช้ Veeam Community Edition แนะนำที่สุด

  • ติดตั้ง Veeam Backup & Replication (ตัวฟรี)
  • เพิ่ม Hyper-V Server เข้าไปในระบบ
  • สร้าง Backup Job และเลือก VM ที่ต้องการ แนะนำให้เปิดฟีเจอร์ Application-aware processing เพื่อให้ Backup ทำงานร่วมกับ Database ภายใน VM ได้สมบูรณ์

🔄 3. การทำ Replication (DR Site ระดับ VM)

โยน VM ไปสแตนด์บายที่ Server อีกเครื่อง (วิธีนี้ง่ายสุด แต่อาจมี Downtime ตอนเปิดเครื่องใหม่)

  1. ฝั่งรับ: เปิด Hyper-V Settings > Replication Configuration > ติ๊ก Enable this computer as a Replica server
  2. ฝั่งส่ง: คลิกขวาที่ VM > Enable Replication... > ใส่ IP เครื่องสำรอง
  3. ตั้งความถี่ (เช่น 5 นาที) แล้วกด Finish

🔌 4. แก้ปัญหา VM มองไม่เห็น Network

ปัญหาเน็ตหลุดสุ่มๆ (ปิด VMQ): คลิกขวา VM > Settings > Network Adapter > Hardware Acceleration > เอาติ๊กออก จาก Enable virtual machine queue

🛢️ 5. ตั้งค่า VM สำหรับ Oracle Database

  • Disk: ต้องใช้ VHDX แบบ Fixed Size เท่านั้น ห้ามขยายเอง
  • RAM: ห้ามติ๊ก Enable Dynamic Memory ให้ใช้ Startup RAM คงที่ (เช่น 8GB)

🛡️ 6. Backup Oracle (ฉบับฟรี)

⚠️ ระวัง: ต้องสั่งรัน RMAN Backup ลงไฟล์ภายใน VM ก่อนเสมอ ห้ามให้ Veeam Snapshot VM ที่รัน Oracle โดยตรง เพราะไฟล์จะ Corrupt

🔗 7. เจาะลึกวิธี Replicate Oracle Database (ไม่มี Data Guard)

สำหรับองค์กรที่ใช้ Oracle Standard Edition และต้องการทำ DR Site (Disaster Recovery) หรือซิงค์ข้อมูล นี่คือ 3 วิธีที่ดีที่สุด พร้อมขั้นตอนการทำครับ

วิธีที่ 1: Log Shipping (Manual Standby) ประหยัดสุด

คอนเซปต์: เครื่องหลักสร้างไฟล์ Archive Log อัตโนมัติ -> เขียน Script (เช่น Python หรือ Batch) ก๊อปปี้ไฟล์ไปวางเครื่องสำรอง -> เครื่องสำรองดึงไฟล์เข้าไปอัปเดต (RPO ข้อมูลหายช่วง 5-15 นาที)

  1. [Primary] เปิดโหมดจัดเก็บ Log: 
    SQL> SHUTDOWN IMMEDIATE;
SQL> STARTUP MOUNT;
SQL> ALTER DATABASE ARCHIVELOG;
SQL> ALTER DATABASE OPEN;
  2. [Primary] เขียน Script ส่งไฟล์: สร้างไฟล์ sync_log.bat ใช้คำสั่ง Robocopy ส่งไฟล์ที่เกิดใหม่ไปหาเครื่องสำรอง แล้วเซ็ต Windows Task Scheduler ให้รันทุก 5 นาที 
    robocopy C:\Oracle\Archive \\DR-Server\SharedArchive /MON:1
  3. [Standby] สร้างเครื่องสำรอง: Copy ไฟล์ Backup ของ Database หลักมาวาง แล้วใช้ RMAN สั่ง Restore ขึ้นมาในสถานะ `MOUNT` (ห้าม Open)
  4. [Standby] เขียน Script รับข้อมูล: เขียน Script ให้ฝั่งสำรองรันคำสั่งนี้ซ้ำๆ เพื่อกวาดไฟล์ Log เข้า Database 
    SQL> RECOVER AUTOMATIC STANDBY DATABASE;
วิธีที่ 2: Oracle SEHA (High Availability) ระบบไม่ล่ม

คอนเซปต์: ทดแทนระบบ RAC ที่แพง โดยใช้ Server 2 เครื่องช่วยกันเฝ้า Shared Storage ก้อนเดียวกัน (เช่น SAN หรือ iSCSI) ถ้าเครื่อง A ดับ เครื่อง B จะดึง Storage ไปรันต่อทันที (ไม่มีข้อมูลสูญหาย 100%)

  1. เตรียม Shared Disk: ใน Hyper-V ให้สร้าง VHDX แบบ Shared Virtual Hard Disk แล้ว Mount เข้าไปใน VM ทั้ง 2 เครื่องพร้อมกัน
  2. ติดตั้ง Oracle Grid Infrastructure (GI): บนเครื่องหลัก เลือกติดตั้งแบบ Standalone Server (Oracle Restart) เพื่อให้ GI จัดการดิสก์ด้วยระบบ ASM (Automatic Storage Management)
  3. ติดตั้ง Database: ลง Oracle DB ไว้ในดิสก์ ASM ที่แชร์กัน
  4. เพิ่ม Node สำรอง: ใช้ฟีเจอร์ของ Clusterware ทำการ Add Node เครื่องที่ 2 เข้าไปในคลัสเตอร์
  5. จำลองสถานการณ์ล่ม: ปิด VM เครื่องหลัก ระบบ GI จะตรวจสอบพบว่าเน็ตเวิร์กขาดหาย และจะทำการ Start Oracle Service ขึ้นมาใหม่บน VM เครื่องที่ 2 ภายในเวลาไม่กี่นาที โดยใช้ IP เดิม (Failover)
วิธีที่ 3: Logical Replication (CDC) ยืดหยุ่นสูงสุด

คอนเซปต์: CDC (Change Data Capture) คือการดักจับข้อมูล "ระดับแถว (Row)" ทันทีที่มีการ Insert/Update แล้วยิงข้อมูลนั้นผ่าน Network ไปลง Database อีกตัว
จุดเด่น: เหมาะกับงานที่ต้องการดึงตารางบางตัวออกไปวิเคราะห์ (Data Warehouse) หรือต่อกับระบบ LIS/Web App โดยไม่ให้ Database หลักทำงานหนักเกินไป รองรับการซิงค์ข้ามยี่ห้อ (เช่น Oracle -> PostgreSQL) โดยใช้ Tool ฟรีอย่าง SymmetricDS

  1. เตรียม Database หลัก: ต้องเปิด Supplemental Logging เพื่อให้ Oracle บันทึกข้อมูลที่ถูกแก้ลงใน Redo Log ให้ครบถ้วน 
    SQL> ALTER DATABASE ADD SUPPLEMENTAL LOG DATA;
SQL> CREATE USER sym_user IDENTIFIED BY pass123;
SQL> GRANT CONNECT, RESOURCE, DBA TO sym_user;
  2. ติดตั้ง SymmetricDS: ดาวน์โหลดและแตกไฟล์ SymmetricDS (เป็น Java Program) ลงใน Server ตัวกลาง หรือตัว Database ก็ได้
  3. ตั้งค่า Engine (ฝั่ง Master): สร้างไฟล์ corp-000.properties ระบุการเชื่อมต่อ Database 
    engine.name=corp-000
db.driver=oracle.jdbc.driver.OracleDriver
db.url=jdbc:oracle:thin:@192.168.1.100:1521:ORCL
db.user=sym_user
db.password=pass123
sync.url=http://192.168.1.100:31415/sync/corp-000
group.id=corp
external.id=000
  4. กำหนด Routing และ Trigger: เข้าหน้า Web GUI หรือรัน Script เพื่อตั้งค่าว่า "ให้จับตาดูตาราง PATIENT_INFO และ LAB_RESULT ถ้ามีการอัปเดต ให้ส่งไปที่ Node ของ DR Site"
  5. ตั้งค่าฝั่งรับ (Target Node): สร้างไฟล์ properties ฝั่งเครื่องรับข้อมูล (เช่น store-001.properties) แล้วสั่ง Registration ให้เครื่องลูกวิ่งมาหาเครื่อง Master
  6. Start Service: สั่งรัน sym --engine corp-000 --port 31415 ระบบจะเริ่มสร้าง Trigger ใน Oracle อัตโนมัติและส่งข้อมูลแบบ Real-time

🏆 คำแนะนำ: เลือกวิธีไหนดีที่สุด?

ขึ้นอยู่กับข้อจำกัดและโจทย์ของระบบในองค์กรคุณครับ:

  • เน้นงบ 0 บาท, ทำง่าย, คุ้นเคยกับ Script:
    ➡️ เลือก วิธีที่ 1 (Log Shipping) เพราะไม่ต้องลงโปรแกรมเพิ่ม ใช้แค่ความรู้ OS และ SQL เล็กน้อย ก็มี DR Site ไว้กู้ชีพได้แล้ว
  • ซีเรียสเรื่อง Data Loss ต้องเป็นศูนย์ (0) และมี Storage กลาง:
    ➡️ เลือก วิธีที่ 2 (Oracle SEHA) เหมาะกับระบบโรงพยาบาลหรือการเงินที่ข้อมูลหายไม่ได้แม้แต่เรคคอร์ดเดียว แต่มีข้อจำกัดคือป้องกันได้แค่ระดับ Server พัง ถ้าระบบไฟดับทั้งตึก (Storage ดับ) ก็ล่มอยู่ดี
  • ต้องการดึงข้อมูลไปทำ Web App หรือระบบ Report โดยไม่กวนระบบหลัก:
    ➡️ เลือก วิธีที่ 3 (CDC - SymmetricDS) วิธีนี้ดีที่สุดหากคุณมีทีมพัฒนา Web (เช่น ทำผ่าน Google Apps Script/Supabase) หรือต้องการแปลงข้อมูลเก่าส่งไปให้ระบบใหม่ๆ แบบ Real-time ครับ

Get-VM | where {$_.State -eq 'Off'} | Start-VM
